## 🟥 P0 — 馬上可做、高 CP 值（低成本、立即有感）

> **✅ 2026-06-13 落地：P0 全四項完成（v4.9.0/v4.9.1）+ P1-3 存進歌庫（v4.10.0）。** 下一波建議 P1-1 AI Vision 或 P2-1 演奏模式。

### P0-1　可疑辨識 token 高亮（OCR 校對神器）✅ 已完成
- **問題**：現在辨識完要逐字用眼睛對原圖找錯字。OCR 其實知道「哪些字它沒把握」。
- **做法**：辨識結果裡「無法解析成和弦、又夾在和弦行中」的 token 標黃底；轉調結果欄同步標。使用者一眼掃到該修哪裡。Tesseract 的 `confidence` 也可一併用（< 60 分的字加虛線底）。
- **成本**：0.5 天　**依賴**：無（資料已在手）　**價值**：⭐⭐⭐⭐⭐

### P0-2　個人慣用調記憶 ✅ 已完成
- **問題**：每首歌每次打開都從原調開始，現場翻歌單要重設。
- **做法**：localStorage 記每首歌（或每段貼上的譜 hash）上次轉到的目標調，下次自動套用。
- **成本**：0.5 天　**依賴**：無　**價值**：⭐⭐⭐⭐

### P0-3　轉調結果下載 + 分享文字 ✅ 已完成（v4.9.1 .txt/分享；v4.10.3 擴充 圖片/PDF）
- **問題**：目前只能「複製」。現場想存檔 / 傳給團員沒有快捷。
- **做法**：加「下載 .txt」（檔名帶歌名+目標調）與「分享」（Web Share API，手機可直接傳 LINE）。
- **成本**：0.5 天　**依賴**：無　**價值**：⭐⭐⭐

### P0-4　數字級數顯示（Nashville Number System）✅ 已完成
- **問題**：吉他老師教學 / 移調思考時，級數（1-4-5-6m）比絕對和弦更直覺。
- **做法**：切換鈕「和弦 ↔ 級數」。依偵測調把 C/F/G/Am 顯示成 1/4/5/6m。教學場景超實用。
- **成本**：1 天（引擎已有 detectKey + 級數計算雛形）　**依賴**：G1　**價值**：⭐⭐⭐⭐（教學加值）

## 🟧 P1 — 中價值、中成本（體驗或品質跳級）

### P1-1　G3b-2 AI Vision 辨識（辨識品質天花板）
- **問題**：反白標籤（`[前奏]→[41%]`）、手機拍照、複雜版面 Tesseract 力有未逮。
- **做法**：Gemini 2.x Flash 視覺模型「看圖直接輸出結構化文字譜」，版面理解接近人工抄譜。**後端代理藏 key**：Supabase Edge Function（免費額度）。UI 做「Tesseract 先免費即時 → 不滿意按『AI 強化辨識』」混合模式，控制每日用量。
- **成本**：1-2 天　**依賴**：Supabase Edge Function　**價值**：⭐⭐⭐⭐⭐
- 📌 *參見既有 skill：supabase-edge-keyless-ai-backend / gemini-free-tier-first*

### P1-2　影像預處理進階
- **問題**：手機拍螢幕會傾斜、有邊框雜物、反白標籤。
- **做法**：(a) 傾斜校正（偵測文字基線角度旋正）；(b) 自動裁切譜面（去掉截圖的瀏覽器框/廣告）；(c) 反白標籤偵測（藍底白字色塊→反轉成黑字白底，解 `[前奏]` 問題）。
- **成本**：1-1.5 天　**依賴**：無（純 canvas）　**價值**：⭐⭐⭐⭐

### P1-3　轉好的譜「存進歌庫 / 變歌曲建議」✅ 已完成（v4.10.0，admin 直接入庫）
- **問題**：辛苦轉好的譜關掉就沒了，沒有沉澱成網站資產。
- **做法**：轉調工具加「存成歌曲建議」→ 串既有 SongSuggestion flow；或 admin 直接「存成正式歌曲」（解析成 lyricBlocks 進 Firestore）。讓 OCR+轉調變成「快速建歌庫」的入口。
- **成本**：1-1.5 天　**依賴**：既有 suggestion / firestore schema　**價值**：⭐⭐⭐⭐⭐（把工具變成內容生產線）

### P1-4　admin 歌詞編輯器內建轉調
- **問題**：admin 入庫一首調性錯的譜時，還是得先去工具轉、再貼回。
- **做法**：T3 規劃的 SongLyricsEditor 內嵌轉調按鈕，入庫前一鍵轉正。
- **成本**：1 天　**依賴**：SongLyricsEditor（T3 Phase 2）　**價值**：⭐⭐⭐

### P1-5　和弦進行級數分析
- **問題**：彈唱者想快速看懂一首歌的「套路」（4536 / 卡農進行）。
- **做法**：分析進行標出級數與常見 pattern 名稱（「這是 6-4-1-5 公式」）。與 P0-4 共用級數引擎。
- **成本**：1 天　**依賴**：G1　**價值**：⭐⭐⭐（教學/學習）

## 🟨 P2 — 進階演奏輔助（現場彈唱深化）

### P2-1　演奏模式（現場彈唱專用畫面）
- **內容**：轉好的譜進入「大字 + 自動捲動（可調速）+ 節拍器（吃 BPM）+ 深色護眼」全螢幕模式。手機架譜台上跟著捲、跟著彈。
- **成本**：2-3 天　**依賴**：G1　**價值**：⭐⭐⭐⭐⭐（這是吉他手最想要的現場場景）

### P2-2　指型多版本 + 左手指法
- **內容**：每個和弦提供「開放 / 封閉 / 簡化」多種按法切換，標左手 1-2-3-4 指序；難和弦給「簡單替代」（F → Fmaj7 / 小橫按）。
- **成本**：2 天　**依賴**：G1 指型字典　**價值**：⭐⭐⭐⭐（新手友善）

### P2-3　分享轉好的譜（連結 / 圖卡）
- **內容**：轉好的譜產生短連結或雜誌風圖卡（沿用 ShareCard 風格），傳給團員/學生。
- **成本**：1.5-2 天　**依賴**：可能需後端存譜　**價值**：⭐⭐⭐

### P2-4　G2 音訊自動採譜（Chordify 式）
- **內容**：丟 mp3 / YouTube 連結 → essentia.js 瀏覽器端 chromagram + 和弦辨識 → 和弦時間軸 → 串轉調。
- **成本**：1-2 天 spike + 後續　**依賴**：G1　**價值**：⭐⭐⭐⭐（差異化武器，但準確度待驗證）
- 📌 *原理研究見 [G1-transpose.md](./G1-transpose.md#-g2--chordify-式自動採譜研究筆記)*

### P2-5　移調夾最佳化建議升級
- **內容**：現有 Capo 建議只列等效位置。升級為「評分推薦」：考慮開放弦數量（越多越好彈）、音域是否適合人聲、避免高把位，給出 ★ 最推薦選項。
- **成本**：1 天　**依賴**：G1　**價值**：⭐⭐⭐

## 🟩 P3 — 生態整合 / 長線投資

### P3-1　工具完全離線可用
- **內容**：OCR 語言檔（~20MB）用 PWA 快取，轉調工具離線也能用。現場沒網路也不怕。
- **成本**：1 天　**依賴**：既有 SW　**價值**：⭐⭐⭐

### P3-2　批次轉調
- **內容**：一次上傳多張譜圖（多段），合併成一首完整譜再轉。
- **成本**：1.5 天　**依賴**：G3a　**價值**：⭐⭐

### P3-3　個人譜庫（收藏轉好的譜）
- **內容**：使用者收藏轉好的譜到個人空間，建立私人歌本。
- **成本**：2-3 天　**依賴**：需登入 / 後端　**價值**：⭐⭐⭐

### P3-4　搜尋結果一鍵進轉調工具
- **內容**：站內搜到的歌 → 詳情頁「在轉調工具開啟」，把既有 lyricBlocks 帶進工具。打通「搜尋 → 轉調」動線。
- **成本**：0.5 天　**依賴**：G1 + SongDetail　**價值**：⭐⭐⭐

## 🎯 建議落地順序（阿凱老師參考）

1. **先清 P0 四連發**（約 2.5 天）— 全部低成本高感受，把現有工具打磨到「順手」：可疑字高亮 → 慣用調記憶 → 下載分享 → 級數顯示。
2. **再做 P1-3「存進歌庫」**（1-1.5 天）— 把 OCR+轉調從「一次性工具」升級成「歌庫生產線」，這是讓工具產生長期價值的關鍵一步。
3. **接著 P1-1 AI Vision**（1-2 天）— 辨識品質一次到位，解掉所有頑固案例（反白標籤、手機拍照）。
4. **最後 P2-1 演奏模式**（2-3 天）— 補完「現場彈唱當下」的最後一塊拼圖，吉他手體驗閉環。

> P2-4 音訊採譜（Chordify）價值高但風險也高（準確度未知），建議當「探索型 spike」獨立排期，別卡在主線上。
