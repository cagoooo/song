## 🚦 建議實施順序

### 第 1 波：清債（v4.7，6-8h）
**先做技術債，否則後續每個 PR 都吃這些坑的虧。**
1. T1 CSS 拆檔（2h）— 開發效率立刻恢復
2. T2 補 ritual modal 測試（6h）— 防回歸

### 第 2 波：高 CP 值產出（v4.8，10-12h）
**做完使用者立刻有感的功能。**
3. D3 PDF 列印版（3-4h）— 演出當下可發紙本節目單
4. T3 SongDetail 串 Firestore（4h）— 解鎖任何歌曲都有真實樂譜
5. D1 期刊期數系統（4-6h）— 「ISSUE №12」不再 hardcode

### 第 3 波：遊戲化深化（v4.9，12-14h）
**讓 5 件套之間連得更深。**
6. C3 徽章擴張（8h）— 從 6 個 → 12-15 個 + 解鎖動畫
7. D5 mood tags（3-4h）— 為 C4 AI 推薦鋪路

### 第 4 波：內容平台（v5.0，14-18h）
**editorial 從工具升級成「平台」。**
8. D2 主理人專欄（6-8h）
9. D4 讀者來信（5-6h）
10. T4 eventId props 預留（4h）— 為下一輪 B2 鋪路

### 第 5 波：進階儀式（v6.0，18-22h）
**真實樂譜 + 影像剪輯。**
11. C1 LRC 歌詞同步（10h）— 演出大螢幕對齊歌詞
12. D6 精選時刻剪輯（8-10h）— 演出後產出 highlight reel

**全部完成估計 ~70-80 小時**，分 5 個版本完成。

## 📝 文件慣例

每份設計文件採同一結構：

1. **TL;DR** — 一段話講完
2. **動機** — 為什麼要做、現況痛點
3. **資料模型** — Firestore schema / TypeScript types
4. **API / Hooks / Components** — 介面定義
5. **UI 草圖** — 文字描述版面
6. **實作步驟** — 編號清單，可當 PR todo
7. **驗收條件** — 怎樣算完成
8. **風險 / 已知坑** — 實作前要知道的雷
9. **估時** — 樂觀 / 預期 / 悲觀
10. **依賴項** — 必須先完成什麼
